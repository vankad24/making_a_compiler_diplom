## Введение

### Актуальность темы
В современных условиях стремительного развития информационных технологий языки программирования играют ключевую роль в создании программного обеспечения различной сложности. 
Популярные языки программирования предлагают различные подходы и инструменты для написания кода, однако зачастую они имеют избыточный синтаксис, высокую сложность освоения, наследуют структурные недостатки предшественников и недостаточную выразительность в определённых предметных областях. Это затрудняет освоение начинающими разработчиками, выпуск новых квалифицированных кадров, усложняет поиск ошибок, увеличивает время разработки и отладки, и создаёт трудности при поддержке приложения.
Постоянное развитие программных систем, рост требований к производительности, безопасности и масштабируемости приводят к необходимости разработки новых решений.

Разработка нового языка программирования позволяет применить совершенно новые подходы в семантическом и синтаксическом анализе для решения практических задач.
Таким образом, анализ современных инструментов разработки языков программирования нацелен на изучение внедрения новых подходов для дальнейшей разработки языка программирования, обладающим такими преимуществами, как: простота использования, хорошая читаемость, широкая функциональность и достаточная выразительность с поддержкой современных подходов программирования и автоматическим поиском ошибок на этапе анализа.

Особую значимость приобретают инструменты разработки языков программирования, такие как генераторы лексических и синтаксических анализаторов, системы построения компиляторов и интерпретаторов, а также средства автоматизации анализа кода. Эти инструменты позволяют существенно сократить время разработки языков, повысить их надёжность и упростить сопровождение.

Актуальность данной работы обусловлена необходимостью анализа возможностей современных инструментов разработки языков программирования, а также изучения подходов к созданию интерпретаторов как одного из наиболее доступных способов реализации языков.

### Анализ существующих языков программирования и их особенностей

Разработка нового языка программирования требует предварительного анализа существующих решений, их преимуществ и недостатков. Такой анализ позволяет выявить ключевые проблемы современных языков и сформулировать требования к инструментам разработки компиляторов и интерпретаторов для устранения недостатков.

Одной из важных задач является исследование возможности выполнения статического анализа программы до её запуска. В частности, представляет интерес выполнение следующих проверок:

- автоматическое определение типов
    
- анализ достижимости кода и полноты ветвлений
    
- возможность реализации различных видов сокращённых конструкций, например, для цикла `for`

Реализация подобных механизмов позволяет проверить возможность реализации и изучить уровень гибкости, которого может достичь создаваемый язык программирования.

#### Классификация языков программирования

Современные языки программирования можно классифицировать по нескольким основным критериям.

**1. По способу выполнения:**

- компилируемые (например, C/C++);
    
- интерпретируемые (например, Python);
    
- смешанные (с использованием виртуальной машины, например, Java).
    

**2. По парадигме программирования:**

- императивные;
    
- функциональные;
    
- логические;
    
- объектно-ориентированные;
    
- мультипарадигменные.
    

**3. По системе типов:**

- статически типизированные и динамически типизированные;
    
- строго типизированные и слабо типизированные.
    

Статическая типизация позволяет выявлять ошибки на этапе компиляции или анализа, что повышает надёжность программ. В свою очередь, динамическая типизация обеспечивает большую гибкость и скорость разработки, но переносит часть ошибок на этап выполнения.

Таким образом, выбор системы типов является компромиссом между безопасностью и удобством разработки.

#### Популярные языки и их особенности

Рассмотрим наиболее распространённые языки программирования и их характерные черты.

- **C/C++** — обеспечивают высокую производительность и контроль над ресурсами, однако обладают сложным синтаксисом. Наличие небезопасных конструкций (например, работа с памятью [3]) приводит к частым ошибкам, а диагностические сообщения компилятора могут быть трудны для понимания [4].
    
- **Java** — предлагает строгую объектную модель, переносимость и хорошо структурированный код. Однако язык характеризуется избыточностью синтаксиса, что увеличивает объём кода [5, 6].
    
- **Python** — отличается простотой и лаконичностью, что делает его популярным для быстрого прототипирования [7]. При этом язык уступает по производительности [8] и не обеспечивает полноценной статической проверки типов [9].
    
- **Go** — сочетает строгий синтаксис и простоту, а также предоставляет удобные средства для разработки параллельных программ. Однако язык не всегда позволяет выразить сложные конструкции компактно [10].
    
- **Rust** — ориентирован на безопасность работы с памятью [11] и предотвращение ошибок на этапе компиляции [12]. Главным недостатком является высокая сложность освоения и трудность реализации комплексных структур.

Отдельно следует отметить, что такие языки, как C/C++, исторически развивались длительное время. Добавление нового функционала часто приводит к усложнению синтаксиса и появлению неоднородных решений, что затрудняет их использование и реализацию компиляторов.

Язык Python, напротив, получил широкое распространение благодаря простоте. Однако отсутствие встроенной строгой статической типизации привело к появлению внешних инструментов, таких как mypy, предназначенных для анализа типов. Это указывает на востребованность подобных механизмов, которые, по возможности, должны быть встроены в сам язык.

### Общая схема интерпретатора
Работа любого языка программирования начинается с считывания текста, представляющего собой исходный код программы, и последующее преобразование текста в удобную для анализа структуру.  Зачастую этой структурой является дерево разбора. Это сложный процесс, поэтому его разбивают на несколько этапов, за которые отвечают разные подпрограммы. В общем случае порядок работы современного компилятора или интерпретатора выглядит следующим образом:
Исходный код, лексический анализ (Lexer), синтаксический анализ (Parser), семантический анализ, генерация исполняемого файла / интерпретация [1, c 32-34].

### Лексический анализ
Первым этапом обработки исходного кода программы является лексический анализ.
Можно сказать, что исходный код представляет собой массив символов. Для более удобной обработки, его необходимо разбить на части. Простое разделение по пробелам не подойдет, так как в программном коде, в отличии от обычного текста, есть арифметические и специальные символы, между которыми может даже не быть пробелов. Поэтому, для правильного разделения входных символов, на практике используют регулярные выражения.
Для примера рассмотрим следующий код:
`int x = 10;`
Данный код отвечает за создание целочисленной переменной `x` со значением `10`.
Разбив данную строку, мы получим:
`'int', 'x', '=', '10', ';'`
В контексте лексического анализа данные последовательности символов называют **лексемы** [1, c 158].
Рассмотрим аналогичный фрагмент кода, который создаёт переменную с плавающей точкой:
`float num = 2.4;`
Если разбить его на лексемы, получим:
`'float', 'num', '=', '2.4', ';'`
В данных двух примерах мы можем заметить сходство: первым пишется тип переменной, затем название, знак равно, числовое значение, знак точка с запятой. Запишем это в более общей форме:
`<тип 'float'> <название 'num'> <знак '='> <число '2.4'> <знак ';'>`
Каждый такой элемент называется **токеном**. Он представляет собой категорию символа и хранит в себе исходный текст (лексему) [1, c 158]. Используя принятые обозначения, которые часто используются на практике и в литературе, запишем это следующим образом:
`<KEYWORD 'int'>, <ID 'x'>, <OPERATOR '='>, <NUMBER '10'>, <KEYWORD ';'>` и
`<KEYWORD 'float'>, <ID 'num'>, <OPERATOR '='>, <NUMBER '2.4'>, <KEYWORD ';'>`.
Зарезервированные в языке символы называют KEYWORD, названия переменных или функций - ID или IDENTIFIER, арифметические знаки OPERATOR, а числовые значения - NUMBER.
Токены могут включать в себя дополнительную информацию [1, c 159], такую как: категорию лексемы, номер строки, номер столбца, имя файла.
Обычно, чтобы разбить исходный текст на набор токенов, используют регулярные выражения (regex) [1, c 168]. Например, `IDENTIFIER` можно представить регулярным выражением `[a-zA-Z_][a-zA-Z0-9_]*`, а NUMBER представить как `[0-9]+`.


##### Лексические ошибки
Ошибка на этапе лексического анализа возникает при несоответствии входного символа ни одному из определений лексем [1, 161].
Например, код:
`string 2s = "Hello world";`
вызовет ошибку, поскольку название переменной (`IDENTIFIER`) не может начинаться с цифры. Также лексические ошибки могут возникать при использовании специфичных символов или кодировок.

##### Заключение
Лексический анализ — это начальный этап обработки, на котором исходный код разбивается на последовательность лексем и преобразуется в набор токенов [1, с 156]. Этот процесс позволяет структурировать текст программы и подготовить его для дальнейшего синтаксического анализа.
За разбиение на токены и поиск ошибок отвечает лексический анализатор (лексер), используя регулярные выражения.
В современных языках программный код лексера обычно не пишут вручную. Для этого используют кодогенерацию на основе заранее прописанной лексической грамматики. Один из самых популярных генераторов лексеров - flex. Он написан на C++, но позже на его основе появились реализации для других языков программирования [1, c 191].

### Синтаксический анализ
**Синтаксический анализ (парсинг)** — это этап обработки исходного кода, на котором набор токенов преобразуется в структурированное представление, которое называется абстрактным синтаксическим деревом (AST).
На этом этапе проверяется соответствие программы правилам грамматики, выявляются ошибки, строится синтаксическое дерево [1, c 252-253].

##### Понятие грамматики
Ещё раз рассмотрим ранее упомянутые примеры:
Набор токенов для строки `int x = 10;`:
`<KEYWORD 'int'>, <ID 'x'>, <OPERATOR '='>, <NUMBER '10'>, <KEYWORD ';'>`
И для строки `float num = 2.4;`:
`<KEYWORD 'float'>, <ID 'num'>, <OPERATOR '='>, <NUMBER '2.4'>, <KEYWORD ';'>`.
Далее для удобства будем записывать `KEYWORD 'float'` как `FLOAT`, `<KEYWORD 'int'> как `INT`, `<OPERATOR '='>` как `EQUAL`, `<KEYWORD ';'>` как SEMICOLON.
Можно заметить, что полученные последовательности довольно схожи. Один из базовых подходов в программировании и в разработке программного обеспечения - это абстракция и обобщение. Здесь мы тоже воспользуемся этим методом. В общем виде инициализация переменной (variable initializaton) будет выглядеть так:
`TYPE ID EQUAL NUMBER SEMICOLON`
Тип переменной - это TYPE, название переменных, как мы уже говорили, принято называть ID, оператор присваивания - EQUAL, число для инициализации - NUMBER, а точка с запятой - SEMICOLON [1, c 260-261].
Большинство элементов здесь представляют собой простые токены, но TYPE является составным, поскольку состоит из FLOAT, либо INT. Запишем для него определение:
`TYPE: FLOAT | INT`
Знак `:` означает 'является', а знак `|` означает 'или'.
В теории формальных языков и грамматик, данная запись называется правилом производства (продукция), где атомарные элементы (токены) называются терминалами, составные - нетерминалами [1, c 258-260]. 
Первый нетерминал, с которого начинается самое первое правило производства, называется начальным символом и обозначается он как S или Start.
Грамматика является набором продукций, задающие правила построения допустимых цепочек символов.

Таким образом, в примере выше, FLOAT и INT являются терминалами, а TYPE - нетерминалом.

Усложним наш пример: 
`int n = 2 + 2 - 1`
Теперь знак точки с запятой в конце является необязятельным, а вместо числа можно написать арифметическое выражение без скобок со знаками плюс или минус.
Тогда фрагмент грамматики для инициализации переменной будет выглядеть следующим образом:

```
VAR_INIT: TYPE ID ASSIGN EXPRESSION LINE_END
TYPE: FLOAT | INT
EXPRESSION: NUMBER MATH_OP EXPRESSION | NUMBER
MATH_OP: PLUS | MINUS
LINE_END: SEMICOLON |
```
VAR_INIT - это продукция инициализации переменной, TYPE - тип, EXPRESSION - арифметическое выражение, MATH_OP - математический оператор плюс или минус, LINE_END - конец строки.
В данном примере нетерминал LINE_END порождается терминалом SEMICOLON или может вовсе отсутствовать. Нетерминал EXPRESSION в своём определении имеет рекурсивный вызов, что является ключевым принципом в построении грамматик и позволяет описывать конструкции произвольной длины.


Как и код лексера, код синтаксического анализатора (парсера) генерируется на основе грамматики, описанных в таком же формате, как было описано в примерах выше. Одним из самых популярных инструментов для генерации парсера является byacc. На его основе также существуют аналоги для других языков программирования, например byacc/j [1, c 363]. Таким образом, создание любого языка сводится к описанию правил производства, то есть к грамматике.

##### Классификация грамматик
В классификации, предложенной Ноаном Хомским, грамматики делят на 4 вида.
###### Неограниченные грамматики
Это общий класс грамматик, где все остальные являются его частными случаями.
Правила имеют вид:
α→β
где:
- α — непустая строка символов
- β — любая строка
Такие грамматики используют редко, Поскольку в таких грамматиках есть возможность поместить несколько нетерминалов в левой части правила производства, они обладают высокой сложностью.


###### Контекстно-зависимые грамматики

Правила имеют вид:
αAβ→αγβ
Где:
- α, β — любая строка символов
- A — нетерминал
- γ — непустая строка
Как следует из названия, применения правил производства в таких грамматиках зависит от окружения. Если говорить более простым языком, происходит замена A на γ. Особенностью является то, что длинна получившейся строки после применения продукции не должна уменьшается.
Данные грамматики всё ещё обладают высокой сложностью, поэтому применяются ограниченно.

###### Контекстно-свободные грамматики
Правила имеют вид:
A→γ
где:
- A — нетерминал
- γ — любая строка символов
Поскольку в левой части производства находится всего один нетерминал, контекстно-свободные грамматики обладают не такой высокой сложностью и легко обрабатываются программно.
Широко применяются для описания синтаксиса языков программирования. [1, c 258]

Пример кс грамматики:
```
Expr → Expr + Term  
Expr → Term  
Term → number
```

###### Регулярные грамматики
Правила имеют вид:
- A→aB
- A→a
где:
- A, B — нетерминалы
- a — терминал
Регулярные грамматики являются наиболее простым классом грамматик.
Широко используются в поиске и шаблонах благодаря регулярным выражениям. На этапе лексического анализа с помощью регулярных выражений описываются идентификаторы, числа, ключевые слова и так далее. 

##### Синтаксические деревья разбора

Дерево разбора - представление исходного кода программы в виде структуры данных "дерево", узлы которого представляют собой нетерминальные и терминальные символы, и соответствуют грамматике языка [2, 115].
Есть два основных вида деревьев разбора:

###### Полное дерево разбора
Полное дерево разбора (Parse Tree, Concrete syntax tree) полностью отражает грамматику языка и содержит все нетерминальные и терминальные символы [2, 115-116].

Пример
Рассмотрим выражение:
2 + 3 * 4
И упрощённую грамматику:
E → E + T | T    
T → T * F | F    
F → (E) | number
Дерево разбора будет содержать все промежуточные нетерминалы:
Картинка.

Полные деревья разбора почти не используются, поскольку содержат много избыточной информации (например, скобки, служебные правила).

###### Абстрактное синтаксическое дерево
Абстрактное синтаксическое дерево (AST, Abstract Syntax Tree) представляет собой упрощённое полное дерево разбора. Оно лишено ненужных деталей, а часть узлов сокращены. Остаются лишь значимые символы, которые в дальнейшем пригодятся для обработки. Такие элементы, как скобки, удаляются, поскольку порядок выполнения и принадлежность символов к конкретному выражению реализуется за счёт древовидной структуры хранения данных. [2, 117]
На практике, при обработке языков используют именно AST, что даёт ряд преимуществ: компактность хранения, ускорение и упрощение обхода дерева, легкость выполнения модификаций. Создание AST реализуют либо методом удаления узлов из полного синтаксического дерева, либо не значимые символы изначально игнорируются на этапе обработки цепочек грамматики.

Пример
Для того же выражения:
2 + 3 * 4

AST будет выглядеть так:
Картинка

##### Виды синтаксических анализаторов

Синтаксический анализатор (парсер) работает на основе **контекстно-свободной грамматики**, получает на вход результат лексического анализа, а в качестве результата выдаёт дерево разбора.
По принципу работы существует два основных вида анализаторов:
###### LL-парсер
Это нисходящий парсер [1, c 283], который:
- читает вход слева направо (**L — left-to-right**)
- строит левый вывод (**L — leftmost derivation**)

Наиболее распространённый вариант — **LL(1)**:
- использует **1 символ предпросмотра (lookahead)**
###### LR-парсер
**LR-парсер** — это восходящий парсер [1, c 302], который:
- читает вход слева направо (**L**)
- строит правый вывод в обратном порядке (**R**)
LR-парсеры работают по принципу:
свёртки (reduce) последовательности символов к нетерминалам. 

Существуют различные модификации:
- LR(0) — простой, но слабый
- SLR(1) — улучшенный
- LALR(1) — компромисс (используется в Yacc)
- LR(1) — самый мощный

##### Синтаксические ошибки
Ошибка на этапе синтаксического анализа возникает при несоответствии входного потока символов ни одному из определений правил производства грамматики [1, c 254-255].
Например, код:
`string s = "Hello" "world";`
вызовет ошибку, поскольку между двумя строками нет знака действия или оператора. 
В другом примере:
`string s = "Hello world"`
можно увидеть ошибку, знакомую многим программистам - пропущен знак точка с запятой.
Стоит отметить, что на этом этапе проверяется только соответствие правилам грамматики, поэтому код `float f = "Hello" / 5;` на данном этапе будет корректным.

##### Заключение
Синтаксический анализатор отвечает за преобразование последовательности токенов в структурированное представление программы — абстрактное синтаксическое дерево. Именно на этом уровне формируется понимание структуры кода в соответствии с заданной грамматикой, а также выявляются синтаксические ошибки.

Грамматика выступает основой языка, позволяя описывать допустимые конструкции через правила производства. 

Выбор конкретного метода синтаксического анализа (LL или LR) зависит от требований к языку и сложности грамматики, однако оба подхода широко применяются в современных программах.

Таким образом, синтаксический анализатор стоит дерево разбора, которое за тем передаёт на следующий этап обработки - семантический анализ.

### Семантический анализ

После построения абстрактного синтаксического дерева следующим этапом является семантический анализ. Его цель — выявить смысловые ошибки и дополнить дерево необходимой информацией посредством его обхода и анализа [2, с 147].

##### Таблица символов
Таблица символов — это структура данных, которая хранит информацию об идентификаторах в программе: имя, тип, значение, ссылка на экземпляр родительского элемента и другие дополнительные атрибуты [2, с 148].

##### Проверка инициализации
Во время обхода дерева разбора, при обработке узла с оператором присвоения значения, анализатор взаимодействует таблицу символов. Если это первая инициализация идентификатора - добавляется новая запись в таблицу символов, если повторная - вызывается ошибка. При присвоении значения к идентификатору проверяется, чтобы значение уже присутствовало в таблице символов, то есть уже было инициализировано ранее [2, с 160].

##### Проверка областей видимости
Область видимости определяет, в каком месте программы и к какому именно идентификатору осуществляется доступ.
Основные виды областей видимости:
- глобальная (за пределами функций)
- локальная (внутри функций)
- блочная (внутри блоков if, while и подобных)
Зачастую допускается объявление идентификаторов с одинаковым именем в разных или вложенных областях видимости.
Один из способов проверки их корректности осуществляется с помощью иерархической структуры в таблице символов. То есть при обращении к идентификатору в таблице символов ищется записать с соответствующим именем, если такой записи не нашлось, то поиск продолжается в родительской таблице [2, с 151].

##### Проверка типов
Проверка типов включает в себя проверку совместимости типов в выражениях, в присвоениях, передаваемых аргументах функции и возвращаемых значений функции.
В выражениях типы данных проверяются, основываясь на типе, указанном в таблице символов и на основе литералов, которые строго задают тип значения. В большинстве языков программирования допускается автоприведение типов, когда итоговое значение отдаётся по приоритету. Например, при сложении чисел с типами данных float и int результат будет float, поскольку у него приоритет больше [2, с 179].

##### Аннотирование AST
После выполнения проверок, для упрощения последующей обработки, в AST добавляется дополнительная информация [2, с 152], такая как: 
- типы выражений
- ссылки на записи в таблице символов
- информация о преобразованиях типов

##### Семантические ошибки
Ошибка на этапе семантического анализа возникает при несоответствии смысловой корректности программы требованиям языка.
К таким несоответствиям относятся:
- несоответствие типов данных
- использование необъявленных идентификаторов
- повторное объявление переменных в одной области видимости
- несоответствие количества или типов аргументов функции
- применение неподдерживаемых операторов

Например, код:
`int s = "Hello world";`
вызовет ошибку, поскольку к переменной с типом int присваивается значение с типом string. 
##### Заключение
Семантический анализ является важным этапом обработки программы, потому как от него зависит преждевременное нахождение смысловых ошибок, связанных с неправильным использованием конструкций языка, таких как несоответствие типов, нарушение областей видимости и некорректное обращение к идентификаторам. Степень сложности обнаруживаемых ошибок, ограничивается только реализацией анализатора и может включать и более сложные сценарии. 

Результатом данного этапа является абстрактное синтаксическое дерево, дополненное информацией необходимой для дальнейшего выполнения программы и её оптимизации.


### Интерпретация
Следующим этапом обработки является использование полученного абстрактного синтаксического дерева. Строго говоря, существует лишь два последующих пути обработки: генерация кода (машинного кода или кода другого языка программирования) и выполнение программы (интерпретация).
В данном исследовании будет рассмотрен процесс интерпретации.
Используя алгоритм обхода дерева в глубину, программа посещает все узлы и выполняет действия в соответствии с типом узла и сохранённой в нём информацией [2, с 215].

Обход дерева чаще всего реализуется рекурсивно. Для каждого типа узла определяется соответствующая логика выполнения:

- **выражения** вычисляются и возвращают значение;
- **операторы присваивания** изменяют состояние таблицы символов;
- **условные конструкции** выбирают ветку выполнения;
- **циклы** повторяют выполнение поддерева до выполнения условия выхода.

В процессе интерпретации ключевую роль играет **таблица символов**. При обработке узлов дерева интерпретатор обращается к этой таблице для чтения и обновления значений. Именно в ней хранится состояние всей программы.


##### Ошибки во время выполнения
В данных класс ошибок попадают все остальные ошибки, которые не удалось выявить на предыдущих этапах. Чаще всего это ошибки, связанные со значениями самих переменных.

Например, код:
```
int a = 24;
int b = a / 2 - 12;
int r = 5 / b;
```
вызовет ошибку "деление на ноль". На этапе семантического анализа подобные ошибки трудно выявить, поскольку они требуют исполнения кода программы для определения значений переменных, но зачастую проверки и вовсе невозможны, поскольку данные приходят из вне.


##### Заключение
Важно отметить, что интерпретатор выполняет программу на ходу, без предварительной генерации машинного кода. Это упрощает реализацию и повышает переносимость, однако уступает компилируемым программам в производительности.

Использование данной среды существенно повышает продуктивность разработки и упрощает сопровождение проекта.

Для реализации лексического анализа используется инструмент JFlex [15]. Он представляет собой генератор лексических анализаторов, аналогичный утилите Lex [16], но ориентированный на Java. Инструмент позволяет описывать правила токенизации с использованием регулярных выражений, автоматически генерируя эффективный код. 

Выбор JFlex обусловлен рядом преимуществ по сравнению с альтернативами:
- в отличие от Lex и Flex, JFlex изначально ориентирован на Java и не требует интеграции с нативным кодом;
- по сравнению с ANTLR [17], JFlex предоставляет более низкоуровневый и предсказуемый контроль над процессом лексического анализа, что важно для учебного интерпретатора;
- генерируемый код отличается высокой производительностью и простотой интеграции;
- в сравнении с ручной реализацией, это уменьшает время разработки и количество ошибок.



Синтаксический анализ реализован с использованием BYACC/J [18]. Данный инструмент является адаптацией классического генератора парсеров YACC [19] для языка Java. Он позволяет задавать контекстно-свободную грамматику языка и автоматически генерировать LALR(1)-парсер. 


Использование BYACC/J обосновано следующими факторами:
- инструмент реализует классический подход LALR(1), который широко применяется в компиляторах и часто используется в литературе;
- по сравнению с ANTLR, BYACC/J использует более традиционную модель восходящего анализа, что делает его ближе к классическим учебным материалам по теории компиляции;
- BYACC/J генерирует компактный и предсказуемый код без сложной рантайм-инфраструктуры;
- по сравнению с ручной реализацией, использование генератора парсеров снижает вероятность ошибок и упрощает разработку.

Альтернативные решения, такие как ANTLR [17] является более функциональным и сложным инструментом, однако он скрывает детали работы парсера. В рамках данной работы предпочтение было отдано BYACC/J, так как он позволяет глубже изучить процесс синтаксического анализа и лучше контролировать его поведение.

Таким образом, выбранные инструменты позволяют эффективно реализовать интерпретатор, обеспечивая удобство разработки, автоматизацию, отладки системы и избавляя от части потенциальных ошибок.

##### Структура программы интерпретатора

Программная реализация интерпретатора языка Aqua организована в виде модульной структуры, разделённой на два основных уровня: фронтенд (анализ исходного кода) и рантайм (выполнение программы). Все исходные файлы располагаются в директории `/src`, что соответствует стандартной практике организации Java-проектов.

Основной файл
Точкой входа в приложение является файл `Main.java`. Данный класс содержит минимальную логику и выполняет делегирование управления методу `main()` класса `j0`. Такой подход позволяет изолировать пользовательский интерфейс запуска программы от внутренней логики интерпретатора и упрощает дальнейшую модификацию архитектуры.

Код в директории frontend (см. Приложение А) отвечает за обработку исходного текста программы и преобразование его в синтаксическое дерево.
- `j0.java` — центральный класс, который координирует процессы: инициализирует лексер, запускает парсер и управляет потоком обработки входных данных.
    
- `Yylex.java` — лексический анализатор, сгенерированный с помощью JFlex. Выполняет разбиение входного потока символов на последовательность токенов.
    
- `Parser.java` — синтаксический анализатор, сгенерированный с использованием BYACC/J. На основе заданной грамматики строит дерево разбора.
    
- `Token.java` — структура, представляющая токен. Содержит тип токена, его текстовое представление и номер строки, что необходимо для диагностики ошибок.
    
- `Tree.java` — реализация узла синтаксического дерева. Включает символ грамматики, список дочерних узлов и связанные токены.
    
- `ParserVal.java` — вспомогательная структура, используемая парсером для передачи значений между правилами грамматики.
    
- `Serial.java` — генератор уникальных идентификаторов, применяемый для маркировки узлов дерева и других объектов.
    
- `Yyerror.java` — модуль обработки синтаксических ошибок, обеспечивающий информативные сообщения при некорректном вводе.
    

Таким образом, пакет `frontend` реализует полный цикл преобразования исходного текста программы в абстрактное синтаксическое дерево.

В директории runtime (см. Приложение Б) находятся классы, которые отвечают за интерпретацию и выполнение.
- `Interpreter.java` — ключевой компонент системы. Выполняет обход синтаксического дерева, проводит семантический анализ и интерпретирует конструкции языка.
    
- `SymbolTable.java` — реализация таблицы символов. Обеспечивает хранение переменных, их значений и поддерживает вложенные области видимости (scope).
    
- `RuntimeValue.java` — универсальная обёртка для значений времени выполнения, содержащая тип и соответствующее значение.
    
- `ValueType.java` — класс с набором констант, определяющих возможные типы данных в языке (например, `int`, `string` и др.).
    
- `PrimitiveHandler.java` — модуль обработки примитивных операций, таких как арифметические вычисления и операции сравнения.
    
- `FunctionHandler.java` — отвечает за обработку вызовов функций, включая передачу аргументов и возврат значений.
    
- `FunctionType.java` — структура, описывающая сигнатуру функции (параметры, тип возвращаемого значения).
    
- `ErrorHandler.java` — компонент обработки ошибок времени выполнения, обеспечивающий корректное завершение программы и вывод диагностической информации.




### Описание языка программирования Aqua

#### Синтаксис и семантика языка

Язык программирования Aqua представляет собой процедурно-ориентированный интерпретируемый язык с синтаксисом, близким к языкам семейства C/Java, но упрощённым для учебных целей. Программа на Aqua состоит из набора операторов и функций, где точкой входа обычно является функция `main`.

Синтаксис языка характеризуется следующими особенностями:

- использование фигурных скобок `{}` для определения блоков кода;
    
- опциональное использование точки с запятой `;` в конце выражений;
    
- возможность записи условий без обязательных круглых скобок;
    
- поддержка однострочных конструкций без явного блока.
    

Идентификаторы формируются по стандартным правилам: начинаются с буквы или символа подчёркивания и могут содержать буквы, цифры и `_`. Язык чувствителен к регистру.

Семантика языка основана на интерпретации абстрактного синтаксического дерева. Все выражения вычисляются последовательно, а управление потоком осуществляется через стандартные конструкции (условия, циклы, функции). Области видимости реализуются с помощью вложенных таблиц символов, что позволяет корректно обрабатывать локальные переменные и параметры функций.

---

#### Типы данных и конструкции управления

Aqua поддерживает ограниченный, но достаточный для исследовательских задач набор типов данных:

- `int` — целые числа;
    
- `float` — числа с плавающей точкой;
    
- `bool` — логические значения;
    
- `intarray` — массивы целых чисел;
    

Типизация статическая: тип переменной задаётся при объявлении или вычисляется автоматически и должен соблюдаться при использовании.

Язык поддерживает стандартный набор операций:

- арифметические (`+`, `-`, `*`, `/`, `%`);
    
- логические (`&&`, `||`, `!`);
    
- операции сравнения (`<`, `>`, `==`, `!=` и др.);
    
- операции присваивания и сокращённые формы (`+=`, `++` и др.).
    

Ключевые конструкции управления:

- **Условные операторы**: `if`, `else if`, `else`;
    
- **Циклы**:
    
    - `while` и `do-while`;
        
    - расширенный `for` с несколькими вариантами синтаксиса (включая диапазоны и шаг);
        
- **Управляющие операторы**: `break`, `continue`, `return`.
    

Функции объявляются с помощью ключевого слова `fn` и могут принимать параметры и возвращать значения. Поддерживается рекурсия.

#### Особенности и отличия от других языков

Язык Aqua обладает рядом особенностей, отличающих его от распространённых языков программирования:

1. **Упрощённый синтаксис**
    
    - необязательные точки с запятой;
        
    - опциональные скобки в условиях;

	- опциональный тип данных при объявлении переменной
        
    - компактные формы записи циклов.
        
2. **Расширенный синтаксис цикла for**
    
    - поддержка нескольких форм записи (короткая, диапазонная, полная);
        
    - возможность задания шага и границ в декларативной форме.
        

3. **Автоопределение типов**
	- Возвращаемый тип функции определяется автоматически

	- Тип при инициализации определяется за счёт литералов
		

По сравнению с Java язык Aqua проще и менее формализован (например, отсутствует обязательная объектная модель). В отличие от Python, Aqua использует статическую типизацию и фигурные скобки вместо отступов. По сравнению с C/C++ отсутствуют указатели и ручное управление памятью.

Ограничения языка включают отсутствие:

- объектно-ориентированного программирования;
    
- обработки исключений;
    
- обобщённых типов (generics);
    
- развитой стандартной библиотеки.

### Реализация лексического анализатора

#### Описание лексем
Правила разбора на лексемы задаются в файле `javalex.l` (см. Приложение А) с использованием регулярных выражений.

Основные группы токенов:

**1. Ключевые слова**

```lex
"int", "float", "if", "else", "while", "for", "fn", "return"
```

**2. Идентификаторы**

```lex
[a-zA-Z_][a-zA-Z0-9_]*
```

**3. Числовые литералы**

```lex
[0-9]+
[0-9]*"."[0-9]*([eE][+-]?[0-9]+)?
```

**4. Строковые литералы**

```lex
\"([^\"]|(\\.))*\"
```

**5. Операторы**

- арифметические: `+`, `-`, `*`, `/`;
    
- логические: `&&`, `||`;
    
- сравнения: `==`, `!=`, `<`, `>` и др.;
    
- составные: `+=`, `++`, `--`.
    

**6. Игнорируемые элементы**

- пробельные символы;
    
- комментарии (`//`, `/* */`).
    

Каждому регулярному выражению соответствует код, который должен выполниться при считывании — вызов метода `scan()` с передачей кода токена или выполнение вспомогательных функций, например, увеличить номер строки.

#### Класс лексера
Лексер в интерпретаторе Aqua реализован с использованием генератора JFlex (см. Приложение В), который на основе регулярных выражений, формирует класс `Yylex.java` (см. Приложение А).
Основным методом класса является `yylex()`. Данный метод считывает символы из входного потока, сопоставляет их с шаблонами регулярных выражений и для каждой лексемы выполняет код, который был указан в файле `javalex.l`.

#### Создание токенов
После перехода из метода `yylex()` в метод `scan()` класса `j0`, начинается создание объекта класса `Token`. В конструктор этого класса передаются: код токена, переданный в виде параметра; текст лексемы, получаемый из метода `Yylex.yytext()`; номер строки - `Yylex.yylineno`. Затем токен передаётся парсеру через поле `j0.par.yylval`.

#### Обработка ошибок
Лексический анализатор включает базовую обработку ошибок. В случае обнаружения неизвестного символа вызывается метод `yyerror` класса `Yyerror`, формирующий сообщение об ошибке.
Пример ошибки:
```
Lexer error: unrecognized character '@' at line 1
```

#### Заключение
Подводя итог, процессы на данном этапе можно описать следующим образом:
Сперва исходный файл считывается как поток символов, лексер последовательно анализирует входной поток, затем распознанные последовательности символов преобразуются в токены и, наконец, полученные токены передаются синтаксическому анализатору.

### Реализация синтаксического анализатора

#### Описание грамматики
Правила синтаксического разбора задаются в файле `j0gram.y` (см. Приложение А). В фигурных скобках после правил прописаны семантические действия, которые указывают, какие из токенов попадут в дерево разбора, а какие будут проигнорированы.

Основные группы конструкций:

**1. Описание блоков кода и методов**
```yacc
Stmt: Block | ';' | ExprStmt | BreakStmt | ReturnStmt
      | IfStmt | IfElseStmt | MethodDecl
      | DoWhileStmt | WhileStmt | ForStmt | LocalVarDeclStmt;

MethodDecl: FN IDENTIFIER '(' FormalParmList ')' Block { ... }
```

**2. Описание переменных и типов**
```yacc
FieldDecl: Type VarDecls { ... }
Type: INT | FLOAT | BOOL | STRING | Name | INTARRAY
```

**3. Операторы и выражения**
- арифметические, логические, сравнения, присваивания, вызовы функций и др.

#### Класс парсера
С помощью BYACC/J (см. Приложение В) на основе файла грамматики автоматически генерируется класс `Parser.java` (см. Приложение А). Ключевым методом является `yyparse()`. Внутри себя данный метод вызывает функцию `yylex()` для получения следующего для обработки токена, а затем сопоставляет значение с заданными правилами производства. Для передачи значений между правилами используется класс `ParserVal`, который, по своей сути, является контейнером для объекта `Token`. Токены помещаются в объекты класса `Tree`, которые реализуют узлы дерева разбора.

#### Обработка ошибок
При обнаружении синтаксической ошибки вызывается метод `yyerror` класса `Yyerror`, формирующий сообщение с указанием строки и типа ошибки.

Пример ошибки:
```
Parser error: unexpected token '}' at line 10
```

#### Заключение
Сформированные на предыдущем этапе токены передаются парсеру, который, используя правила производства указанные в файле грамматики `j0gram.y`, строит синтаксическое дерево и сообщает об ошибках.
Класс `j0.java`, в свою очередь, выполняет роль координатора, вызывая методы обоих анализаторов.

### Реализация семантического анализатора

Полученное на предыдущем этапе абстрактное синтаксическое дерево подлежит анализу на предмет ошибок и для вычисления служебной информации. Этот функционал реализует метод `semantic()` класса `Interpreter` (см. Приложение Б). В качестве параметров он принимает дерево разбора и глобальную область видимости, в которой уже заданы некоторые значения, например, встроенные функции. Работа семантического анализатора, начинается с вызова `analyzeBlock` для корня дерева разбора.
Данный метод рекурсивно обходит дерево, выполняя ряд операций:
  - Объявления переменных: добавляет новые записи в `SymbolTable`, проверяет их на повтор.
  - Присваивания: проверяет существование переменной, типы, допустимость операций.
  - Анализирует выражения: вызывает функцию `analyzeExpr()`, которая вычисляет тип результата, вызывает обработчики операций.
  - Вызовы функций: проверяет наличие, количество и типы аргументов, создает новую область видимости.
  - Вложенные блоки: создает вложенные таблицы символов.
  

В основе реализации `SymbolTable` (см. Приложение Б) стоит структура данных словарь, где ключом является имя переменной, которому соответствует её текущее состояние. В Java такая структура объявляется как `HashMap<String, RuntimeValue>`. `RuntimeValue.java` представляет собой универсальный контейнер для значений разных типов (int, float, bool, массивы, функции), который содержит тип переменной и само значение.
Помимо этого таблица символов поддерживает вложенность областей видимости через ссылку на родителя, а также реализует следующие методы:
  - `addVar(String name, RuntimeValue value)`: добавляет переменную, проверяет на повтор.
  - `getVar(String name)`: ищет переменную в текущей и родительских областях.
  - `contains(String name)`: проверяет наличие переменной.
По завершении обработки, ссылка на `SymbolTable` сохраняется в соответствующем узле самого синтаксического дерева.

Класс `PrimitiveHandler` (см. Приложение Б) отвечает за проверку типов операндов, автоматическое приведение к нужному типу и реализацию базовых операций над типами, а именно: арифметические, логические, сравнение.

Обработка функций осуществляется в классе `FunctionHandler` (см. Приложение Б). При вызове функции он находит её в таблице символов, проверяет аргументы, создает новую область видимости, выполняет тело функции. Также класс имеет методы для обработки встроенных функций, такие как: добавление их в область видимости и вызов их реализации, написанной на Java.

#### Обработка ошибок
Обработка ошибок находится `ErrorHandler` (см. Приложение Б). При возникновении нештатной ситуации вызывается один из методов класса, который, в свою очередь, выбрасывает исключение, обработка которого находится на уровень выше, в классе `j0`.

#### Заключение
Семантический анализатор гарантирует корректность программы на уровне типов, областей видимости, операций и вызовов функций, предотвращая ошибки времени выполнения.

### Выполнение кода

Метод `eval()` класса `Interpreter` отвечает за вычисления узлов дерева и выполнение исходной программы.  
Как и в семантическом анализе, здесь используется схожий принцип обхода дерева разбора, но вместо анализа узлов дерева происходит их исполнение.
Выполняет различные действия в зависимости типа обрабатываемого узла:
  - **Блоки**: создает новую область видимости, интерпретирует все инструкции по порядку.
  - **Присваивания**: вычисляет выражение, сохраняет результат в таблице символов.
  - **Выражения**: вызывает `evalExpr` которое, возвращает вычисленное значение.
  - **Вызовы функций**: вызывает `FunctionHandler.eval`, передает аргументы, создает новую область видимости для локальных переменных.

#### Работа с функциями
Пользовательские функции в классе `FunctionHandler` обрабатываются следующим образом:
  - При вызове создается новая область видимости.
  - Аргументы копируются в локальные переменные.
  - Последовательно интерпретируются инструкции тела функции.
  - Возвращаемое значение передается обратно.

Для встроенных функций вызывается реализация, написанная на языке Java.
Перечень встроенных функций и их назначение:
 - print(...) — выводит значения аргументов в одну строку через пробел.
 - println(...) — выводит значения аргументов в одну строку через пробел и переводит строку.
 - printChar(int code) — выводит символ по его числовому коду.
 - readInt() — читает целое число с ввода.
 - readFloat() — читает число с плавающей точкой с ввода.
 - newIntArray(int size) — создает новый массив целых чисел заданной длины.
 - readCharsAsIntArray() — читает все строки ввода, возвращает массив кодов символов.
 - readChar() — читает один символ с ввода, возвращает его числовой код.
 - getElement(intarray arr, int index) — возвращает элемент массива по индексу.
 - setElement(intarray arr, int index, int value) — устанавливает значение элемента массива по индексу.
 - getLen(intarray arr) — возвращает длину массива.
 - getArg(int index) — возвращает аргумент командной строки как массив кодов символов.
 - readFile(intarray path) — читает файл по пути (массив символов), возвращает содержимое как массив кодов символов.


#### Обработка ошибок
Для обработки ошибок, как и в семантическом анализе, используется класс `ErrorHandler`.

#### Заключение
Данный этап является финальным в последовательности обработки исходного кода. Процесс интерпретации начинается с корневого узла дерева, во время которого рекурсивно выполняются все конструкции программы, изменяется состояние таблиц символов и в поток вывода выводится результат. 
Стоит отметить, что на данном этапе не производится проверка инициализации переменных, областей видимости и типов. Благодаря семантическому анализу обеспечивается корректная работа программы без необходимости вычислительных затрат на проверку ошибок.

### Тесты
В ходе разработки интерпретатора были реализованы юнит тесты на языке Java, используя библиотеку JUnit5. Код тестов находятся в файле `tests/main/InterpreterTests.java` (см. Приложение Г). Тестируемые скрипты на языке aqua находятся в папке `tests/files/`.

Для избегания повторения кода в тестах реализованы две служебных функции:
- **test_success(String filename, Object var_value):**
  1. Загружает и запускает скрипт из файла.
  2. После интерпретации извлекает значение переменной `res` из глобальной области видимости.
  3. Проверяет, что код завершения — 0, и результат совпадает с ожидаемым.

- **test_fail(String filename, String error_msg):**
  1. Запускает скрипт и ожидает ошибку во время выполнения.
  2. Проверяет, что сообщение об ошибке содержит ожидаемое сообщение.
#### Итог
Тесты автоматически проверяют как успешные сценарии, так и обработку ошибок, ускоряя разработку программы.

### Интеграция со средой разработки
Для повышения удобства разработки на языке Aqua была реализована интеграция со средой разработки Sublime Text [20]. Это позволило обеспечить базовую поддержку языка на уровне редактора и упростить процесс написания и запуска программ.

В рамках интеграции разработаны следующие компоненты (см. Приложение Е):

- файл подсветки синтаксиса `Aqua.sublime-syntax`, реализующий разметку ключевых конструкций языка, включая ключевые слова, литералы, операторы и идентификаторы;
    
- файл настроек комментариев `Aqua - Comments.tmPreferences`, обеспечивающий корректную работу однострочных и многострочных комментариев;

Файл `Aqua.sublime-build` интегрирует интерпретатор Aqua в стандартный механизм сборки Sublime Text, благодаря чему запуск программ осуществляется через встроенные средства среды без необходимости обращения к командной строке. Поддерживается запуск для windows, linux и debug режим.
Стоит отметить, что для корректной работы интерпретатор Aqua должен быть установлен в системе (см. Приложение В).

Реализованная интеграция обеспечивает базовую поддержку языка Aqua в редакторе: автоматическое распознавание `.aqua` файлов, подсветка синтаксиса, подсказки и система запуска программы. Это делает разрабатываемый язык более удобным и приближенным к использованию полноценных языков программирования.

        this.name = name;
        this.parent = null;

        assigned_vars = new HashSet<>();
        is_inside_function = false;
    }
    public SymbolTable(String name, SymbolTable parent){
        this.name = name;
        this.parent = parent;
        variables = new HashMap<>();
        assigned_vars = new HashSet<>();
        is_inside_function = parent.is_inside_function;
    }

    public RuntimeValue getVar(String name){
        if (variables.containsKey(name)) return variables.get(name);
        if (parent==null) return null;
        return parent.getVar(name);
    }

    public boolean contains(String name){
        return variables.containsKey(name);
    }
    public void addVar(String name, RuntimeValue runtimeValue) {
        if (variables.containsKey(name))
            ErrorHandler.interpreterError("Redeclaration of name" + name);
        variables.put(name, runtimeValue);
    }

    public void setVar(String name, RuntimeValue value){
        var current_value = variables.get(name);
        if (current_value!=null){
            if ( current_value.type!=value.type)
                ErrorHandler.interpreterError("Assign type mismatch for name: "+ name);
            variables.put(name, value);
        } else if (parent != null) parent.setVar(name, value);
        else ErrorHandler.interpreterError("Unknown name: "+ name);
    }

    boolean isInitialized(String var_name){
        if (assigned_vars.contains(var_name))return true;
        else {
            if (parent == null)return false;
            else return parent.isInitialized(var_name);
        }
    }

public class Interpreter {

    public static void semantic(Tree root, SymbolTable global_scope) {
        analyzeBlock(root, global_scope);

    public static void analyzeBlock(Tree node, SymbolTable scope) {
        if (node == null)return;
        switch (node.sym) {
            case "LocalVarDecl" -> {
                declareVar(node.kids[0].tok, node.kids[1].tok.text, scope);
            }
            case "Assignment" -> {
                String var_name = node.kids[0].tok.text;
                RuntimeValue result;
                String op = node.kids[1].tok.text;
                if (!op.equals("=")) {
                    String s = switch (op) {
                        case "+=", "-=", "++", "--" -> "AddExpr";
                        case "*=","/=" -> "MulExpr";
                        default -> null;
                    };
                    Tree add_value;
                    if (op.equals("++") || op.equals("--")) add_value = new Tree("token", -1, new Token(Parser.INTLIT, "1", -1));
                    else add_value = node.kids[2];
                    result = analyzeExpr(new Tree(s, -1, new Tree[]{new Tree("token", -1, node.kids[0].tok), new Tree("token", -1, new Token(-1, op.substring(0, 1), -1)), add_value}), scope);
                }else{
                    result = analyzeExpr(node.kids[2], scope);
                }
                var v = scope.getVar(var_name);
                if (v == null) scope.addVar(var_name, result);
                else checkType(node.kids[0].tok, var_name, v.type, result.type);
                scope.assigned_vars.add(var_name);
            }
            case "MethodCall" -> {
                checkMethodCall(node, scope);
            }
            case "IfStmt", "IfElseStmt" -> {
                var r = analyzeExpr(node.kids[0], scope);
                checkType(node.tok, "if stmt condition", ValueType.BOOL, r.type);

                var if_block = node.kids[1];
                var if_scope = new SymbolTable("if", scope);
                analyzeBlock(if_block, if_scope);
                if (node.sym.equals("IfElseStmt")) {
                    var else_block = node.kids[2];
                    var else_scope = new SymbolTable("else", scope);
                    analyzeBlock(else_block, else_scope);

                    // set vars as assigned for if-else
                    for (String el : if_scope.assigned_vars) {
                        if (else_scope.assigned_vars.contains(el)) {
                            scope.assigned_vars.add(el);
                        }
                    }
                }
            }
            case "WhileStmt", "DoWhileStmt" -> {
                if (!node.sym.equals("DoWhileStmt")) {
                    var r = analyzeExpr(node.kids[0], scope);
                    checkType(node.tok, "while stmt condition", ValueType.BOOL, r.type);
                }
                var while_block = node.kids[1];
                var while_scope = new SymbolTable("while", scope);
                analyzeBlock(while_block, while_scope);

                // set vars as assigned for do-while
                if (node.sym.equals("DoWhileStmt")) {
                    scope.assigned_vars.addAll(while_scope.assigned_vars);
                    var r = analyzeExpr(node.kids[0], while_scope);
                    checkType(node.tok, "while stmt condition", ValueType.BOOL, r.type);
                }
            }
            case "ForStmt" -> {
                var for_block = node.kids[1];
                var for_scope = new SymbolTable("for", scope);
                var for_header = node.kids[0];
                if (for_header.sym.equals("ForShort")){
                    var r = analyzeExpr(for_header.kids[0], scope);
                    checkType(node.tok, "for initialization", ValueType.INTEGER, r.type);
                }else{
                    // "ForNormal"
                    var for_var_init = for_header.kids[0];
                    var var_name = for_var_init.kids[0].tok.text;
                    declareVar(new Token(Parser.INT,"",0),var_name, for_scope);
                    if (for_var_init.rule == 1224){
                        var r = analyzeExpr(for_var_init.kids[1], for_scope);
                        checkType(node.tok, "for initialization", ValueType.INTEGER, r.type);
                    }
                    for_scope.assigned_vars.add(var_name);

                    var r = analyzeExpr(for_header.kids[2], for_scope);
                    checkType(node.tok, "for limit", ValueType.INTEGER, r.type);

                    //"ForFull"
                    if (for_header.sym.equals("ForFull")){
                        r = analyzeExpr(for_header.kids[3], for_scope);
                        checkType(node.tok, "for step", ValueType.INTEGER, r.type);
                    }

                }

                analyzeBlock(for_block, for_scope);
            }
            case "ReturnStmt" -> {
                if (!scope.is_inside_function) ErrorHandler.syntaxError(node.tok,"return statement outside a function");
                if (node.kids.length != 0) analyzeExpr(node.kids[0], scope);
            }
            case "BlockStmtsOpt", "BlockStmts", "Block" -> {
                for (Tree kid : node.kids) {
                    analyzeBlock(kid, scope);

                }
            }
            default -> {
            }
        }

    public static RuntimeValue evalBlock(Tree node, SymbolTable scope) {
        if (node == null || node.sym.equals("token") && node.tok.text.equals(";")) return null;
        switch (node.sym) {
            case "LocalVarDecl" -> {
                declareVar(node.kids[0].tok, node.kids[1].tok.text, scope);
            }
            case "Assignment" -> {
                String var_name = node.kids[0].tok.text;
                RuntimeValue result;
                String op = node.kids[1].tok.text;
                if (!op.equals("=")) {
                    String s = switch (op) {
                        case "+=", "-=", "++", "--" -> "AddExpr";
                        case "*=","/=" -> "MulExpr";
                        default -> null;
                    };
                    Tree add_value;
                    if (op.equals("++") || op.equals("--")){
                        add_value = new Tree("token", -1, new Token(Parser.INTLIT, "1", -1));
                        add_value.calculated_value = new RuntimeValue(ValueType.INTEGER, 1);;
                    }
                    else add_value = node.kids[2];
                    result = evalExpr(new Tree(s, -1, new Tree[]{new Tree("token", -1, node.kids[0].tok), new Tree("token", -1, new Token(-1, op.substring(0, 1), -1)), add_value}), scope);
                }else{
                    result = evalExpr(node.kids[2], scope);
                }
                var v = scope.getVar(var_name);
                if (v == null) scope.addVar(var_name, result);
                else scope.setVar(var_name, result);
            }
            case "MethodDecl" -> {
                // already declared in global scope
            }
            case "MethodCall" -> {
                processMethodCall(node, scope);
            }
            case "IfStmt" -> {
                return processIfElse(node.kids[0], node.kids[1], null, scope);
            }
            case "IfElseStmt" -> {
                return processIfElse(node.kids[0], node.kids[1], node.kids[2], scope);
            }
            case "WhileStmt" -> {
                return processWhile(node.kids[0], node.kids[1], scope);
            }
            case "DoWhileStmt" -> {
                return processDoWhile(node.kids[0], node.kids[1], scope);
            }
            case "ForStmt" -> {
                return processFor(node.kids[0], node.kids[1], scope);
            }
            case "ReturnStmt" -> {
                if (node.kids.length == 0) return new RuntimeValue(ValueType.NONE);
                return evalExpr(node.kids[0], scope);
            }
            case "BlockStmtsOpt", "BlockStmts", "Block" -> {
                for (Tree kid : node.kids) {
                    var result = evalBlock(kid, scope);
                    if (result != null) return result;
                }
            }
            default -> {
                ErrorHandler.notImplementedError(node.sym + " " + node.tok);
            }
        }
        return null;
    }

    public static RuntimeValue evalExpr(Tree node, SymbolTable scope) {
        switch (node.sym) {
            case "token" -> {
                Token t = node.tok;
                if (t.code == Parser.IDENTIFIER)
                    return scope.getVar(t.text);
                else
                    //PrimitiveHandler.literalToValue(node.tok);
                    return node.calculated_value;
            }
            case "MethodCall" -> {
                return processMethodCall(node, scope);
            }
            case "AddExpr", "MulExpr", "RelExpr", "EqExpr", "CondAndExpr", "CondOrExpr" -> {
                RuntimeValue r = node.calculated_value;
                if (r!=null && r.value != null) return r;

                var left = evalExpr(node.kids[0], scope);
                var op = node.kids[1].tok.text;
                var right = evalExpr(node.kids[2], scope);
                return processOperator(left, op, right);
            }
            case "UnaryExpr" -> {
                var op = node.kids[0].tok.text;
                var right = evalExpr(node.kids[1], scope);

                switch (op){
                    case "-" -> { return processOperator(new RuntimeValue(right.type, 0), "-", right); }
                    case "!" -> { return processOperator(new RuntimeValue(ValueType.BOOL, false), "==", right); }
                    default -> {}
                }

            }

        }
        return null;
    }

    static RuntimeValue processIfElse(Tree condition_expr, Tree if_block, Tree else_block, SymbolTable scope) {
        var condition = evalExpr(condition_expr, scope);
        if ((boolean) condition.value) {
            return evalBlock(if_block, new SymbolTable("if_stmt", scope));
        } else if (else_block != null) {
            if (else_block.sym.equals("IfElseStmt")) return evalBlock(else_block, scope);
            else return evalBlock(else_block, new SymbolTable("else_stmt", scope));
        }
        return null;
    }

    static RuntimeValue processWhile(Tree expr, Tree block, SymbolTable scope) {
        var condition = evalExpr(expr, scope);
        while ((boolean) condition.value) {
            var result = evalBlock(block, new SymbolTable("while_stmt", scope));
            if (result != null) return result;
            condition = evalExpr(expr, scope);
        }
        return null;
    }

    static RuntimeValue processDoWhile(Tree expr, Tree block, SymbolTable scope) {
        SymbolTable do_while_scope = new SymbolTable("do_while_stmt", scope);
        var r = evalBlock(block, do_while_scope);
        if (r != null) return r;
        var condition = evalExpr(expr, do_while_scope);
        while ((boolean) condition.value) {
            do_while_scope = new SymbolTable("do_while_stmt", scope);
            r = evalBlock(block, do_while_scope);
            if (r != null) return r;
            condition = evalExpr(expr, do_while_scope);
        }
        return null;
    }
